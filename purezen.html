Case Study

PureZen

A full-stack, AI-powered spa booking platform on AWS — natural-language concierge for guests, an operations portal for staff, and a fully serverless backend underneath.

A concept build for a fictional spa brand, architected and built solo on AWS and deployed serverless on Lambda, S3, and CloudFront, with Anthropic's Claude powering the conversational layer. Live demo: purezen.stephsimmons.dev ↗ · Code: GitHub ↗

The problem

Spa booking is a transactional domain where wrong answers have real cost — a concierge that invents availability or double-books is worse than no concierge at all. The goal: a natural-language booking experience that feels effortless for guests, but is grounded in live data and can never hallucinate a slot that doesn't exist.

PureZen handles the full lifecycle — browse services, check availability, book, reschedule, cancel — through conversation, backed by an admin portal for staff scheduling and operational insight.

Architecture

A fully serverless, event-driven design on AWS — no servers to patch, automatic scaling, and pay-per-request economics that scale to zero when idle.

Highlights

  • Serverless compute — AWS Lambda runs the API per request; nothing to patch or keep warm, automatic scaling under load, and zero cost when idle.
  • Global delivery — the static frontend ships from S3 behind CloudFront, with TLS and edge caching for low-latency access anywhere.
  • Resilient data — DynamoDB with point-in-time recovery (35-day window), GSIs, TTL-based session expiry, and conditional writes that make double-booking impossible.
  • Security by design — least-privilege IAM execution roles (no credentials in code), bcrypt-hashed passwords, UUID session tokens rotated on login with server-side invalidation.
  • Observability — structured CloudWatch logs and metrics across the Lambda and data layers.

AI design — grounded, not guessing

The MCP pattern

PureZen separates the model (text generation), tools (structured functions like get_available_slots(), book_slot()), and data (live DynamoDB). The model never answers from training knowledge — it only formats data returned by tool calls. It cannot invent availability because it only sees what the database returns. The LLM's role shifts from "answer provider" to "response formatter" — the correct role in a transactional system, and one that makes every response auditable back to a specific query.

Regex-first intent routing

Even with fast managed inference from Anthropic's Claude, the design routes ~80% of messages through a deterministic regex classifier in under 200ms, invoking the model only for genuinely ambiguous input. Keeping the model a fallback rather than the front door means lower latency, predictable cost, and a clear audit trail for every booking action.

Engineering decisions & trade-offs

Why deterministic logic over the LLM wherever possible?

The most reliable features used the least AI. Personalized recommendations (category-frequency scoring) and admin trend queries are deterministic formatters reading live data — faster, cheaper, and more reliable than a model for questions with exact answers. The principle: decide when not to use AI before writing code that uses it.

From self-hosted to managed inference

An earlier iteration self-hosted a small open model on a CPU instance — workable, but CPU-bound and slow under concurrency. Moving the conversational layer to Anthropic's Claude removed that bottleneck: fast, reliable inference with no GPU or instance management, which also unlocked the move to a fully serverless backend. Pairing managed inference with the regex-first router keeps the common paths instant and the model's cost scoped to the cases that actually need it.

Treating infrastructure as code

An environment reset mid-development was costly because the system existed only in its running state. Adding a requirements.txt, a one-command deploy.sh with health check, and seed scripts cut full rebuild time to ~15 minutes — a hard lesson in reproducibility.
